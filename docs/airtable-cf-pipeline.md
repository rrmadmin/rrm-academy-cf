# Airtable → Cloudflare Pipeline Architecture

> Reusable pattern for syncing Airtable CMS data to Cloudflare infrastructure. Used by Courses pipeline, applicable to Library, FAQs, Commentary, and any future content type.

## Overview

```
Airtable (CMS / admin UI)
    │
    ▼
fetch-{domain}-data.mjs          ← Node.js build script
    │
    ├── Structured data → src/data/{domain}.json    (consumed by Astro SSG)
    ├── Attachments     → R2 bucket (rrm-assets)    (permanent CDN URLs)
    └── Binary assets   → R2 bucket (rrm-assets)    (audio, images, etc.)
    │
    ▼
Astro Build (astro build)
    │
    ▼
Cloudflare Pages (static HTML)
    │
    ├── Pages Functions → D1 (runtime data: comments, enrollment, progress)
    └── R2 public URL   → PDFs, audio served via CDN
```

## The Pattern

### 1. Airtable as Single Source of Truth

Airtable is the authoring layer. Content editors (Brian, Gianna) work exclusively in Airtable's UI. They never touch code, JSON, or Cloudflare directly.

**Airtable provides:**
- Structured content (text fields, selects, linked records)
- Binary attachments (PDFs, images) via Attachment fields
- Relational links between tables (Course → Module → Lesson)
- Status fields for publish control (`Published`, `Draft`, `Archived`)

**Airtable does NOT provide:**
- Runtime user data (enrollment, progress, comments → D1)
- Permanent file URLs (attachment URLs expire in ~2 hours → R2)
- Static HTML (generated by Astro at build time)

#### The Green Base / Yellow Base Pattern (Library)

For sensitive data like the research library, a two-tier base architecture provides data isolation:

```
Library "green base" (appyZWo2G7iByXCgZ)     ← master record, full data
    │                                            106 fields on BIFID alone
    │  Airtable Base-to-Base Sync (one-way)
    │  Controlled by single select field
    │  + Airtable automation
    ▼
⚡️ Library "yellow base" (app78UTVdeFph9qhL)  ← curated public subset
    │                                            only fields safe to publish
    │  fetch-data.mjs (build script)
    ▼
articles.json → Astro → rrmacademy.org/library
```

**Why two bases:**
- **Data isolation** — Green base holds proprietary enrichment data, AI training data, internal notes, and fields that must never leak to the public web. Yellow base receives only the curated subset.
- **One-way firewall** — Airtable's base-to-base sync is read-only downstream. Edits in yellow base cannot propagate back to green base. Master data is safe.
- **Selective publishing** — A single select field on green base controls which records sync to yellow base. Not every record goes public.
- **Scale independence** — Green base can grow to hold far more data (enrichment metadata, raw PubMed data, internal classifications) without affecting the public-facing pipeline.

**Current state:** `fetch-data.mjs` currently points to the green base (`appyZWo2G7iByXCgZ`). This should migrate to point to the yellow base (`app78UTVdeFph9qhL`) once the base-to-base sync is fully configured, completing the data isolation.

> This pattern can be replicated for any domain where the master data contains sensitive fields: create a "yellow" public-facing base, sync the safe subset via Airtable automation, and point the fetch script at yellow.

#### Editorial Commentary uses a similar pattern

The Editorial Commentary Blog base uses Airtable automations with a single select field to control which posts are published, matching the same publish-gate approach as the Library pipeline.

### 2. Fetch Script (the sync engine)

Each content domain has a standalone Node.js script: `src/lib/fetch-{domain}-data.mjs`

**Responsibilities:**
1. Fetch all records from Airtable API (with pagination + retry on 429s)
2. Transform Airtable field names to clean camelCase JSON
3. Resolve linked records into nested structures
4. Download attachments from temporary Airtable URLs
5. Upload attachments to R2 for permanent CDN URLs
6. Write the final JSON to `src/data/{domain}.json`

**Invocation:**
```bash
AIRTABLE_PAT=xxx node src/lib/fetch-{domain}-data.mjs
```

**Key design decisions:**

| Decision | Rationale |
|----------|-----------|
| Standalone script, not Astro integration | Can run independently of build, in CI, or locally |
| Filter by Status field | Only `Published` records go to production |
| Exponential backoff on 429 | Airtable rate limits at 5 req/sec |
| `_rawAttachments` → `attachments` two-phase | Separate Airtable fetch from R2 upload for clean error handling |
| Attachment keyed by Airtable ID | `courses/{stepId}/{attId}.{ext}` — replace in Airtable = new R2 object |

### 3. R2 for Binary Assets

**Bucket:** `rrm-assets` (ENAM region)
**Public URL:** `https://pub-4af88159ce884265baba8fb4f3470625.r2.dev`
**Binding:** `ASSETS` in `wrangler.toml`

**Why R2, not `public/` directory:**
- Binary files don't bloat the Pages deployment artifact
- R2 objects persist independently of deploys (no re-upload on every build)
- Same bucket serves PDFs (courses), audio (podcast), and future assets
- Custom domain possible later (`assets.rrmacademy.org`)

**Upload method:** `wrangler r2 object put` via `execFileSync` in the fetch script. Uses the same auth as wrangler pages deploy — no extra credentials needed.

**Key path pattern:** `{domain}/{recordId}/{attachmentId}.{ext}`

### 4. D1 for Runtime Data

**Database:** `rrm-auth` (UUID: `22742c9c-77fa-4344-abda-7e7e8b0da9de`)
**Binding:** `DB` in `wrangler.toml`

D1 stores data that changes at runtime (user actions), NOT content managed in Airtable:

| Table | Purpose | Source |
|-------|---------|--------|
| `user` | Authentication | Runtime (signup) |
| `session` | Login sessions | Runtime (login) |
| `enrollment` | Course enrollment records | Runtime (enroll action) |
| `step_progress` | Lesson completion tracking | Runtime (mark complete) |
| `lesson_comment` | Lesson discussion threads | Runtime (post comment) |

**Rule:** If it's authored by an admin → Airtable. If it's created by a user action → D1.

### 5. Astro Static Generation

Astro reads `src/data/{domain}.json` at build time and generates static HTML pages. The JSON is the contract between the fetch script and Astro.

**Build command:** `astro build && npx pagefind --site dist`
**Deploy:** CF Pages auto-deploys from git push to `main`

### 6. Pages Functions (API layer)

`functions/api/` provides runtime endpoints that read/write D1. These run as Cloudflare Workers on the Pages project.

```
functions/api/
├── auth/         → signup, login, logout, verify email, reset password
├── courses/      → enroll, progress, comments
├── billing/      → subscription management
├── contact/      → contact form submission
└── survey/       → endo survey responses
```

## Applying to New Content Domains

To add a new Airtable → CF pipeline (e.g., Library, FAQs, Events):

### Step 1: Identify the Airtable structure
- Which base? Which tables?
- What are the linked record relationships?
- Which fields are content vs. metadata?
- Are there attachment fields?

### Step 2: Create the fetch script
Copy the pattern from `fetch-courses-data.mjs`:
```javascript
// src/lib/fetch-{domain}-data.mjs

// 1. Define table IDs and field arrays
const TABLE_ID = 'tblXXX';
const FIELDS = ['Title', 'Status', ...];

// 2. fetchTable() — reuse the same paginated fetcher
// 3. transform{Record}() — map Airtable fields to JSON shape
// 4. assemble() — resolve linked records into nested structure
// 5. syncAttachmentsToR2() — if attachments exist
// 6. Write to src/data/{domain}.json
```

### Step 3: Add npm script
```json
{
  "scripts": {
    "fetch-{domain}": "node src/lib/fetch-{domain}-data.mjs",
    "fetch-all": "node src/lib/fetch-data.mjs && ... && node src/lib/fetch-{domain}-data.mjs"
  }
}
```

### Step 4: Create Astro pages
Pages import from `src/data/{domain}.json` and use `getStaticPaths()` for dynamic routes.

### Step 5: Add runtime API (if needed)
If users interact with this content (comments, saves, progress), add Pages Functions that read/write D1.
