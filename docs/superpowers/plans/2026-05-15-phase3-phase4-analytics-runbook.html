1. Pre-flight checks (run before either phase)

Five minutes. Confirms the code-side foundation is healthy before touching CF / GA4 dashboards.

  1. Confirm all Phase 1 + 2 PRs are merged to main 2 min 
    cd ~/iCode/projects/rrm-academy-cf
git fetch origin
git log origin/main --oneline -10 | grep -E "client-analytics-phase[12]"
# Should show 6 commits: phase1, phase2a, phase2b, phase2c, phase2d, phase2e
  2. Confirm /api/track is live and 204-ing 1 min 
    curl -X POST -sS https://rrmacademy.org/api/track \
  -H "Content-Type: application/json" \
  -d '{"event":"cta_click","params":{"id":"runbook-test","page":"/"}}' \
  -w "HTTP %{http_code}\n"
# Expect: HTTP 204 (or 503 if GA4 env vars unset — fix that first)
  3. Confirm GA4 Realtime is receiving events from /api/track 2 min

    Open GA4 → property rrmacademy.org → Reports → Realtime → "Event count by Event name". Should show page_view (constantly), and once you trigger the curl in step 2, cta_click arrives within ~10s.

    If cta_click doesn't show up: GA4 env vars (GA4_MEASUREMENT_ID, GA4_API_SECRET) may not be set in CF Pages. Verify via op read + the admin endpoint's 503 response shape.

Phase 3Zaraz GTM disable + soak

manual (CF dashboard) ~30 min initial, +5 min daily during 7-day soak, +10 min final risk: medium (one-week dashboard dip expected)

Goal: stop Zaraz from injecting GTM at /analytics/. Eliminates 587 KiB of JS, 3 CSP violations, and the duplicate page_view counts that currently inflate GA4 sessions/users by ~2×.

Approach: disable first (reversible in 60 seconds), soak for 7 days, then delete if no regressions. Do NOT delete first.

3.1 Inventory current Zaraz config (record what's there before changing anything)

  1. Open CF dashboard → Zaraz 3 min

    Navigate to: dash.cloudflare.com → account → Zaraz → property rrmacademy.org → Tools.

    Screenshot the entire list. Most likely there's just one: Google Analytics 4 with measurement ID G-TSWRY7XLR0. There may also be Cloudflare-managed entries (Speed Insights, Loading config) — leave those alone.

  2. Inventory triggers 3 min

    Click the GA4 tool. List every trigger it has (Pageview, Click, Form submit, Custom, etc.). Save the screenshot — this is your rollback reference.

    If there are custom triggers firing into GA4 that do not have an equivalent server-side or Phase 2 client beacon, surface them to Brian before disabling. Default: there are none (Zaraz is GTM-vanilla here).

3.2 Disable the GA4 tool (soft pause — reversible in 60s)

  1. Toggle the GA4 tool from "Enabled" → "Disabled" 1 min

    Click the GA4 tool row → toggle the enable switch off. Do not click "Delete" yet. Click "Save" / "Publish" so the change reaches production.

  2. Verify /analytics/ stops returning the 587 KiB GTM bundle 2 min 
    curl -sI https://rrmacademy.org/analytics/ | head -3
# If Zaraz GA4 is the only tool: 404 or 200 with tiny body (Zaraz returns
#   a stub script that does nothing).
# If other Zaraz tools still active: 200 with smaller body than before.

# Hard byte count:
curl -s https://rrmacademy.org/analytics/ | wc -c
# Pre-change: ~587000 bytes. Post-change: <50000 bytes or 404.
  3. Verify GA4 Realtime still shows page_views 2 min

    Open GA4 Realtime. page_view count should continue flowing — those come from server-side middleware (Measurement Protocol), not Zaraz. If page_view goes to zero, you've broken the server pipeline somehow (unlikely from a Zaraz change — investigate immediately and re-enable Zaraz GA4).

  4. Verify CSP errors clear 3 min

    Open the homepage in a clean browser (or incognito + DevTools Console). The previously-recurring CSP violations:

    • googletagmanager.com/gtag/js script blocked
    • analytics.google.com/g/s/collect connect-src blocked
    • stats.g.doubleclick.net/g/collect connect-src blocked

    All three should be gone. PSI on the homepage should now show Best Practices ≥ 92 (was hitting 92 ceiling from these violations).

What you'll see in GA4 over the next 24h: sessions + users count will drop by approximately 30–50%. This is correct — Zaraz was double-counting because both it AND middleware were firing page_view for the same browser visit. The post-disable number is the truth. Annotate this in GA4 (Admin → Data display → Annotations) so future-you doesn't think the site died.

3.3 Soak window (7 days)

  1. Daily check (5 min/day for 7 days) 35 min total

    Once per day during the soak:

    • GA4 Realtime: page_view still flowing? cta_click / scroll_depth / search_submit / faq_expand / glossary_lookup / theme_toggle / video_play / share_click / copy_citation also arriving?
    • GA4 Reports → Engagement → Pages: page-view trend looks roughly half of pre-disable but flat day-over-day? (If it crashes to near-zero, something's broken.)
    • PSI homepage: Performance + Best Practices stable or improved (not regressed)
    • Console errors page: any new CSP violations or third-party-origin warnings?
  2. CHANGELOG entry 5 min

    Add an entry to docs/plans/backlog.md noting:

    ### 2026-05-15: Zaraz GTM disabled

Reason: replaced by first-party /api/track (spec
docs/superpowers/specs/2026-05-15-client-analytics-spec.html).

Expected impact:
  * GA4 sessions/users counts drop ~30-50% (de-duplication of
    page_view that was firing from both Zaraz GTM and middleware MP).
  * The new number is correct. Compare YoY only after 2026-05-22.
  * Best Practices score should improve from 92 → 100 (CSP errors clear).
  * 587 KiB of unused JS removed from /analytics/.

Rollback (60s): re-enable the GA4 tool in CF Zaraz dashboard.

3.4 Final removal (after 7-day soak passes)

  1. Delete the GA4 tool from Zaraz 2 min

    If the 7-day soak showed no regressions: CF dashboard → Zaraz → Tools → GA4 → trash icon → confirm.

  2. If Zaraz has no other tools, disable Zaraz on the zone 2 min

    Cuts the /analytics/ route entirely + removes the Zaraz client loader from every page. If you're keeping any Cloudflare-managed entries, skip this step.

  3. Verify /analytics/ now 404s 1 min 
    curl -sI https://rrmacademy.org/analytics/ | head -1
# Expect: HTTP/2 404
  4. Re-run PSI on homepage 2 min

    Confirm: Performance ≥ 95, Best Practices = 100, "Reduce unused JavaScript" no longer flags the 187 KiB /analytics/ payload.

  5. Close the loop in the analytics spec 2 min

    Update docs/superpowers/specs/2026-05-15-client-analytics-spec.html §8 "Zaraz Removal Plan" — mark step 6 complete, add the date.

Phase 4GA4 dashboard configuration

manual (GA4 dashboard) parts API-able with admin scope ~4–6 hrs over the first week risk: low (dashboard config — no production code touched)

Goal: turn the event stream into useful reports. Register custom dimensions so the spec's 13 personas are queryable, mark conversions so GA4 highlights them, build 13 saved audiences for segmentation, and assemble 20 funnel + 4 path explorations for ongoing analysis.

Sequencing: dimensions first (events without dimensions are unfiltered noise), then conversions (so daily reports surface them), then audiences (which depend on dimensions), then funnels (which depend on conversions).

4.2 Mark 8 conversions (~10 min)

GA4 → Admin → Events → find each event → toggle "Mark as conversion" on:

  • sign_up — top-of-funnel account creation
  • generate_lead — newsletter / endo survey / course waitlist / free enrollment
  • begin_checkout — donation, subscription, or paid course intent
  • purchase — completed donation or paid enrollment
  • pdf_download — high-intent: guide PDFs are deep funnel
  • copy_citation — academic intent (RIS file consumed by Zotero/Mendeley)
  • video_complete — course lesson finished (natural end, not "Mark Complete")
  • scroll_depth WHERE depth = 100 AND article_type ∈ {research, commentary} — full read

The last one (scroll_depth) requires creating a Modify event rule: rename scroll_depth with depth=100 + article_type in [research, commentary] to a new event name like full_read, then mark full_read as the conversion. Alternative: don't mark scroll_depth as a conversion and treat 100% scroll as an engagement metric only.

4.5 Build 4 path explorations (~30 min)

GA4 → Explore → "Path exploration" template. For each, set "Starting point" = the relevant trigger event or page; let GA4 surface what users do next.

  1. AI arriver paths — starting point: session_start with audience filter "AI search arrivers". Reveals which pages AI agents send users to most + what those users do next.
  2. IG arriver paths — same shape, audience "IG arrivers". Validates Naomi's @napro_fertility_surgeon → site funnel.
  3. Donor paths (pre-conversion) — ending point: purchase with lead_source = donation. Backward-walks the last 5 events. Tells you which content/CTAs precede a donation.
  4. Engaged reader paths — starting point: scroll_depth = 75 with audience "Engaged readers". Where do deep-read users go after the article?

4.6 Cohort retention (~10 min)

  1. Build a cohort exploration 10 min

    Explore → Cohort exploration → Cohort inclusion: session_start. Cohort granularity: monthly. Calculation: standard. Returning criteria: any event. Save as "Monthly cohort retention".

    This needs the cohort_date dimension (registered in §4.1) to label cohorts by their first-touch month. Without it you can still cohort by signup date — but cohort_date lets you cohort by arrival, not signup, which is the meaningful comparison.

5. Weekly funnel review ritual

Per CXL (paraphrased): "People happily tag events for micro actions but then never actually do micro-funnel analysis." This ritual is the difference between an instrumentation project and an analytics practice.

Calendar block: 15 min, weekly, same day every week (Brian's pick — Monday morning or Friday wrap). On the appointed time:

  1. Open the GA4 home with the 5–7 pinned funnels
  2. For each pinned funnel: glance at the conversion rate, note any step that dropped > 10 percentage points vs prior week
  3. For the worst-dropped step: spend 5 min in Path Exploration on it — what are the most common next actions for users who didn't continue?
  4. Note one finding in a running log (Notion, a markdown file, whatever). Even "nothing surprising this week" is a valid log entry.
  5. Once per month: spot-check the 13 audience counts. Any that grew > 50% or shrunk > 30% week-over-week deserves an "is this real?" look at the underlying trigger event.

If a funnel exposes an actionable change (e.g. "donate CTA in hero converts 3× better than donate CTA in footer"), it becomes a Phase 2f-style code change — add a new data-track-cta or move a button, ship, soak, re-measure.
