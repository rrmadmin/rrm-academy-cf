1Goal

Get rich behavior + conversion tracking in GA4 without loading any third-party scripts in the browser, without widening CSP, and without the 588 KiB Zaraz/GTM payload at /analytics/. Replace the current half-broken hybrid (server-side MP for conversions + Zaraz GTM for the rest) with a single coherent pipeline.

3Architecture

┌─────────────────────────────────┐ │ Page HTML │ │ <script type="module"> │ │ import {track} from │ │ '/src/scripts/track.ts' │ ← 0-dep, ~1 KiB minified │ track('cta_click',{id:'x'}) │ │ </script> │ └──────────────┬──────────────────┘ │ │ navigator.sendBeacon('/api/track', {event, params}) ▼ ┌─────────────────────────────────┐ │ functions/api/track.js │ │ • Rate-limited (60/min/IP) │ │ • Event-name allowlist │ │ • Param sanitization │ │ • Fire-and-forget GA4 MP │ │ • Mirror to AE for AE queries │ └──────────────┬──────────────────┘ │ ├──→ GA4 (via existing sendGA4Event) │ └──→ CF Analytics Engine (worker_events dataset)

Why both GA4 and Analytics Engine?

  • GA4 is what Brian already opens; engagement reports, funnel analysis, retention — familiar tooling.
  • Analytics Engine is queryable from /api/admin/seo and the SEO monitor worker (5-min granularity, free up to 10M rows/day). Gives us raw event log without GA4's sampling. Already in use for survey events. Tiny incremental cost.

Mirroring is a 5-line addition; not having AE means we'd lose visibility once GA4 dashboards aggregate.

Why not just gtag()?

4/api/track Endpoint Design

File: functions/api/track.js

Methods: POST (event submission), OPTIONS (CORS preflight)

Auth: none required (anonymous behavior tracking is by design)

Rate limit: 60 events / minute / IP via checkRateLimit(). Same limit as survey/event.js. Lower than the cost of writing to AE+GA4 per request.

Request shape (validated)

{
  "event": "cta_click",          // string, allowlist (see §6), max 40 chars
  "params": {                     // object, max 25 keys, max 100 chars per value
    "id": "donate-hero",
    "page": "/",
    "value": 100                  // optional numeric (for value-bearing events)
  }
}

Response: 204 No Content on accept, 400 invalid shape, 429 rate-limited, 503 with body { error: 'service_unavailable' } if GA4 env vars missing. (Use the majority pattern from articles.js/ask.js — not the older { ok: false, error: '…' } shape from survey/event.js.)

Validation rules

  • event must match ^[a-z][a-z0-9_]{0,39}$ AND appear in the allowlist
  • params must be a plain object with 1–25 keys
  • Each param key matches ^[a-z][a-z0-9_]{0,39}$
  • Each param value is string (max 100 chars), number (finite, in [−1e9, 1e9]), or boolean. Arrays/objects rejected.
  • Reserved param names ignored (page_location, page_referrer, engagement_time_msec — those come from server)

Side effects on accept

  1. waitUntil(sendGA4Event(env, request, body.event, body.params)) — relays to GA4 with server-derived session_id, source, medium. Wrapped in waitUntil() because sendGA4Event returns a promise.
  2. env.EVENTS.writeDataPoint({ blobs: [...], doubles: [...], indexes: [event] }) — mirrors to AE for raw queries. Called directly, NOT in waitUntil() — it's synchronous (no promise) and queues internally.

Failure modes

  • GA4 5xx / network error: silent (caught in _ga4.js), AE still gets the event
  • AE not bound: silent, GA4 still gets the event
  • Both fail: still return 204 (analytics failures must never block UX)
Security gate: added to scripts/guard.mjs manifest — the file gets hashed alongside other Phase 4/5 endpoints. Rate-limit must be present. Allowlist must be present.

5Client Helper

File: src/scripts/track.ts

type TrackParams = Record<string, string | number | boolean>;

const ENDPOINT = '/api/track';
const DEBUG = (typeof window !== 'undefined') && window.location.search.includes('debug_track=1');

export function track(event: string, params: TrackParams = {}): void {
  if (typeof navigator === 'undefined') return;
  const payload = JSON.stringify({ event, params });

  if (DEBUG) console.log('[track]', event, params);

  // sendBeacon preferred: survives page unload, no CORS preflight, fire-and-forget.
  // Falls back to keepalive fetch for browsers without sendBeacon (none in current
  // matrix, but the cost is one if-check).
  const blob = new Blob([payload], { type: 'application/json' });
  if (navigator.sendBeacon?.(ENDPOINT, blob)) return;

  fetch(ENDPOINT, {
    method: 'POST',
    body: payload,
    keepalive: true,
    headers: { 'Content-Type': 'application/json' }
  }).catch(() => {});
}

// Convenience wrapper for outbound link instrumentation. Auto-extracts hostname.
export function trackOutbound(event: string, href: string, extra: TrackParams = {}): void {
  let host = '';
  try { host = new URL(href, location.href).hostname; } catch {}
  track(event, { href, host, ...extra });
}

Bundle treatment: Astro bundles src/scripts/track.ts once when imported by any <script> block; CF Pages caches the hashed /_astro/track.*.js for 1 year. Pages that don't import it pay zero bytes.

Import pattern (page-level)

<script>
  import { track } from '../scripts/track.ts';
  document.getElementById('donate-cta')?.addEventListener('click', () => {
    track('cta_click', { id: 'donate-hero', page: location.pathname });
  });
</script>

For widely-reused events (scroll depth, outbound links), a thin src/scripts/track-auto.ts module is imported once in BaseLayout.astro, auto-instrumenting anything with a data-track-* attribute. See §7.

10Privacy & Compliance

  • No PII in event params. Allowlist enforces no email, name, user_id as param keys. Query strings are never sent — only query_length. The endpoint strips any param key matching /email|user|name|password|token|cookie|address|phone|ssn/i defensively before forwarding.
  • Client ID is anonymous hash of IP+UA (existing getClientId in _ga4-source.js). No cookies set by /api/track itself.
  • Do-Not-Track honored. Client helper checks navigator.doNotTrack === '1' and no-ops.
  • EU consent: since data flow is first-party and no third-party scripts run, the ePrivacy bar is lower than gtag. We document this in /privacy-policy/ (separate copy update).

11Sampling & Cost

GA4 free tier: 10M events/month. CF Analytics Engine: 10M write_data_points/month free.

At current ~10k unique visitors/month and ~10 events/visit, we're at ~100k events/month — 1% of either ceiling. No sampling needed.

If traffic grows 100×, add client-side sampling in track.ts: if (Math.random() > 0.5) return; gated by a config flag. Conversion events stay 100% (already server-side).

15.8 Mistakes to avoid (CXL warnings)

Direct CXL quotes (paraphrased) for the recurring traps:

  • One giant funnel. "Don't cram everything into one giant funnel — you'll end up with a mess of averages that hides everything that matters." Each funnel in §15.4 is narrow on purpose.
  • Tracking pageviews / time-on-site as goals. "A common mistake is to track pageviews or time on site as a goal. These aren't business goals." Our key events list (§15.3) is all business outcomes — signup, lead, checkout, purchase, video_complete, full read.
  • Unsegmented funnel analysis. "Looking at an unsegmented funnel is almost pointless. Always segment by device, new vs returning, traffic source, and other meaningful cuts." Every persona in §15.1 doubles as a default funnel segment.
  • Linear funnel assumption. "Users don't move linearly. They go back, open new tabs, revisit later." GA4 funnel explorations support open funnels; for the donor + AI arriver journeys we'll use open funnels with multi-step lookback windows.
  • Sample-size ignorance. "Don't draw conclusions from tiny funnel samples." First 30 days post-launch, funnels with <100 sessions get a warning banner in the (eventual) Looker Studio dashboard.
  • Percentage without absolute. "A step might have a great conversion rate but almost no volume. Check both conversion rate and absolute counts." Every funnel report shows N + %.
  • Broken tracking. "A surprisingly common issue is broken tracking. If your data is wrong, your funnel analysis is worthless." Mitigated by: ?debug_track=1 console output, smoke test in CI, weekly GA4 Realtime check, AE raw event log as crosscheck.
  • Top-of-funnel quality blindness. "If your top of the funnel is full of low-intent traffic, your funnel conversion will look terrible." The AEO arrival funnel (§15.4 #18) is specifically designed to surface this — if AI-sent traffic doesn't engage, the source isn't worth the SEO effort.
  • Defining micro-funnels too broadly. "Micro-funnels are meant to be narrow and specific. If your micro-funnel has ten different paths, it's not a micro-funnel anymore." The content→email and engaged-visitor micro-funnels (§15.4 #16, #17) have strict step definitions.
  • Tracking micro-steps but never analyzing them. "People happily tag events for micro actions but then never actually do micro-funnel analysis." Phase 4 includes a weekly 15-min review ritual; otherwise we're collecting data we never use.

16Proof Gates

Modeled on the existing fact-pipeline gates (G1–G5) and payment-pipeline gates (PG1–PG4). Naming: AG (Analytics Gate). All deterministic — static analysis, no network in --quick mode. Wired into:

  • Pre-commit hook (hooks/pre-commit) — runs npm run gates:analytics:check when any tracked file changes. Quick, no network. Blocks commit on fail.
  • CI deploy (.github/workflows/deploy.yml) — runs npm run gates:analytics on every deploy. Blocks deploy on fail.
  • Manual — node scripts/gates/validate-analytics-pipeline.mjs --gate AG3 for single-gate debugging.
IDGateWhat it prevents
AG1 Endpoint contract functions/api/track.js must:
  • Import checkRateLimit from ../auth/_shared.js AND call it on POST
  • Import sendGA4Event from ../_ga4.js (no inline fetch(google-analytics.com))
  • Import ALLOWED_CLIENT_EVENTS, REQUIRED_PARAMS, PII_REGEX from ./_track-events.js
  • Export onRequestPost AND onRequestOptions
  • Return CORS_HEADERS on every response
Prevents a future PR from quietly reimplementing the GA4 call inline or skipping rate limit.
AG2 Allowlist coverage Every track('event_name', …) call site in src/ with a string literal event name must have event_name ∈ ALLOWED_CLIENT_EVENTS. Catches typos that would 400 in production (track('cta_clck')) before they ship. Known blind spot: dynamic event names from data-track-* attributes in track-auto.ts are not statically resolvable; runtime validation in /api/track catches those instead.
AG3 Server/client event separation Server-only events (page_view, sign_up, signup_from_ask, generate_lead, begin_checkout, purchase) must not appear in ALLOWED_CLIENT_EVENTS. Prevents double-counting if client and server both fire the same event.
AG4 Required params satisfied Every track(name, {…}) call site must include all keys in REQUIRED_PARAMS[name]. Catches track('cta_click', {}) missing the required id + page. Static-parses object literals.
AG5 PII regex intact _track-events.js exports PII_REGEX. The regex source must contain all of: email, user, name, password, token, cookie, address, phone, ssn. A future PR diluting this regex trips the gate.
AG6 UTM convention Any string literal in src/ or functions/ containing ?utm_ or &utm_ must match: lowercase, underscores only (no hyphens or spaces), ASCII-only. Catches utm_campaign=Newsletter Monthly May 2026 (uppercase, spaces) before it pollutes dashboards.
AG7 No third-party analytics scripts Grep src/ + functions/ for script-loading or fetch() to: googletagmanager.com, www.google-analytics.com (outside _ga4.js), stats.g.doubleclick.net, connect.facebook.net. Each must appear ZERO times. (www.google-analytics.com in _ga4.js is the one allowed exception — that's the MP endpoint we proxy through.)
AG8 CSP lockdown CSP_VALUE in functions/_middleware.js must NOT include googletagmanager.com, analytics.google.com, stats.g.doubleclick.net, connect.facebook.net. Catches accidental "I'll just add it to CSP" loosening.
AG9 Track helper exclusivity No raw fetch('/api/track', …) or sendBeacon('/api/track', …) outside src/scripts/track.ts. Forces every emission through the validated helper.
AG10 Conversion completeness Every event in §15.3's conversion list must have at least one call site: server-side via sendGA4Event(env, request, '<name>', …) OR client-side via track('<name>', …). Catches conversion-list drift (someone removes the last call site but forgets to remove the dashboard conversion flag).
AG11 Bundle size After build, dist/_astro/track.*.js bundle size ≤ 2.0 KiB minified+gzipped. Trips on helper bloat. track-auto.ts separately capped at 3.5 KiB.
AG12 Custom dimension parity Every dimension name in §15.2 (registered in GA4) must appear as a param name in at least one track(), sendGA4Event(), or buildSourceParams() call site. Warn-only (not fail) — GA4 has dimensions that fill from default params (page_location, session_id) we don't manually pass. Surfaces drift, doesn't block deploy.

What the gates don't cover

Honest limits — these need eyeballs:

  • GA4 dashboard state. Gates can't see whether conversions are marked, audiences are built, or funnels exist in the GA4 UI. The Phase 4 checklist (§15.9) is the manual handoff.
  • Whether the events are useful. Gates verify we collect what we said we'd collect. They don't verify the data answers business questions. That's the weekly review ritual.
  • Cross-domain consistency. If library.rrmacademy.org or a future subdomain also emits events, they should use the same allowlist. Out of scope this phase.
  • Sampling effects. GA4 sampling above thresholds isn't visible at code level. Use AE raw event log as crosscheck.
