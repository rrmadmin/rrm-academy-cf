# `src/lib/` Quality Baseline — Design

**Date:** 2026-04-20
**Scope:** `src/lib/` (16 files, 2,327 lines — build-time data fetchers and shared TS utilities)
**Status:** Proposed
**Author:** Brian + Claude

## 1. Purpose

Stand up an AI-driven code-quality pipeline for the `src/lib/` directory of `rrm-academy-cf`. This first pass is **instrumentation only**: no new tests, no refactors, no CI gates. The goal is to produce a reproducible baseline report so we can see — for every file and every function — how we stand today on four dimensions:

1. **Line coverage** (how much code the existing tests exercise)
2. **Mutation score** (how well tests detect injected faults)
3. **CRAP score** (cyclomatic complexity × uncovered-risk; surfaces big, untested functions)
4. **Dependency rule violations** (cycles, layering breaks)

With that baseline in hand, a follow-up spec will set thresholds, write tests to hit them, and wire the pipeline into CI. This spec explicitly stops short of that.

## 2. Non-Goals

- Writing new unit tests
- Refactoring `fetch-courses-data.mjs` or any other file
- Setting coverage/mutation thresholds
- Gating CI or deploys on any metric
- Touching `functions/api/`, `src/components/`, or `src/pages/`
- Introducing Gherkin/Cucumber (deferred to a later spec scoped to `functions/api/`)

## 4. Toolchain

| Concern | Tool | Why |
|---|---|---|
| Line/branch coverage | **c8** | Native Node, zero-config for ESM, works with `node --test` (the project's existing runner). No Jest/Vitest migration needed. |
| Mutation testing | **Stryker-js** (`@stryker-mutator/core`) with `@stryker-mutator/api` and the `node-test` runner | Industry standard; supports ESM; per-file reports. |
| Cyclomatic complexity | **ESLint `complexity` rule** (set to report-only) + **`eslint-plugin-sonarjs`** for `cognitive-complexity` | Already have ESLint; no extra toolchain. SonarJS cognitive-complexity is often more actionable than classic CC. |
| CRAP score | **Custom `scripts/quality/crap.mjs`** | No maintained npm package. Formula is trivial: `CRAP(f) = CC(f)² × (1 − coverage(f))³ + CC(f)`. Reads c8 JSON + ESLint JSON output, joins on file+function, emits `reports/quality/crap.json` and a sorted markdown table. |
| Dependency rules | **`dependency-cruiser`** | De-facto standard. Config file lets us encode rules like "no cycles" and "`.ts` utilities must not import build-time `.mjs` fetchers". |

Everything runs locally via npm scripts. No hosted services, no CI changes in this pass.

## 5. Architecture

All quality tooling lives under `scripts/quality/` so it's isolated from build/deploy scripts:

```
scripts/quality/
├── coverage.mjs        # thin wrapper: `c8 --reporter=json --reporter=text node --test test/*.test.js`
├── mutation.mjs        # `stryker run` with src/lib-scoped config
├── crap.mjs            # joins coverage.json + eslint-complexity.json → crap.md + crap.json
├── deps.mjs            # `depcruise src/lib --config .dependency-cruiser.cjs`
├── dashboard.mjs       # runs all four, writes docs/quality/BASELINE.md
└── lib/
    ├── load-coverage.mjs     # parse c8 JSON
    ├── load-complexity.mjs   # run ESLint programmatically, parse results
    └── render-markdown.mjs   # shared table renderer
```

Reports land in `reports/quality/` (gitignored — these are artifacts, regenerate them):

```
reports/quality/
├── coverage/           # c8 HTML + JSON
├── mutation/           # stryker HTML + JSON
├── crap.json
├── crap.md
└── deps.json
```

The **one committed artifact** is `docs/quality/BASELINE.md` — a human-readable snapshot generated by `dashboard.mjs`, checked into git so we can diff quality over time.

## 6. npm Scripts

Added to `package.json`:

```json
{
  "scripts": {
    "quality:coverage":  "node scripts/quality/coverage.mjs",
    "quality:mutation":  "node scripts/quality/mutation.mjs",
    "quality:crap":      "node scripts/quality/crap.mjs",
    "quality:deps":      "node scripts/quality/deps.mjs",
    "quality:all":       "node scripts/quality/dashboard.mjs"
  }
}
```

Each is independently runnable (for iteration speed). `quality:all` runs them in order and regenerates the baseline doc.

## 7. CRAP Formula

Standard definition (Chidamber/Kemerer-style, popularized by Alberto Savoia):

```
CRAP(f) = CC(f)² × (1 − cov(f))³ + CC(f)
```

Where:
- `CC(f)` = cyclomatic complexity of function `f` (from ESLint)
- `cov(f)` = fraction of statements in `f` covered by tests, 0.0–1.0 (from c8)

Interpretation (industry convention):
- CRAP ≤ 5  → healthy
- CRAP ≤ 30 → acceptable
- CRAP > 30 → danger zone (refactor or test)

Our baseline report will bucket every function into these three bands and list all red-band functions at the top of `BASELINE.md`.

## 8. Dependency-cruiser Ruleset (seed)

`.dependency-cruiser.cjs` starts minimal, targeting only `src/lib/`:

1. **`no-circular`** — warn on any import cycle.
2. **`no-orphans`** — warn on files with zero importers *and* zero importees, excluding entry-point fetchers.
3. **`ts-must-not-import-fetchers`** — error if any `*.ts` file imports a `fetch-*.mjs` (TS utilities are for runtime code; fetchers are build-time only).
4. **`no-external-except-allowlist`** — warn on imports outside `src/lib/`, `node_modules`, or Node builtins.

Scoped strictly to `src/lib/`. We'll expand to the rest of the codebase later.

## 9. `BASELINE.md` Format

Sections in order:

1. **Summary table** — one row per file: lines, coverage %, mutation score %, max CRAP, dep violations
2. **CRAP red list** — every function with CRAP > 30, sorted descending, with `file:line`
3. **Coverage gaps** — files with <50% coverage
4. **Mutation survivors** — top 20 surviving mutants (actionable for test authoring)
5. **Dependency violations** — rule + file + direction
6. **Generated** — timestamp and git SHA for traceability

All auto-generated. Hand edits will be overwritten — the file carries a `<!-- generated by dashboard.mjs — do not hand-edit -->` banner at the top.

## 10. Error Handling

- Each script exits non-zero on tool failure (bad config, missing output) but **does not fail on "bad" scores** — thresholds are out of scope.
- `dashboard.mjs` tolerates any single tool failing: it produces the report with a "⚠ tool failed" note for that section, so a broken Stryker install doesn't block the coverage report.
- Reports are deterministic given the same input: no timestamps inside tables, only in the footer. This keeps git diffs of `BASELINE.md` meaningful.

## 11. Testing This Pipeline

Meta-question — does quality-tooling need its own tests? Scoped answer for this pass: **no formal unit tests**, but `crap.mjs` gets a fixture-based smoke test:

- `test/quality-crap.test.js` — feeds synthetic coverage + complexity JSON through `crap.mjs`, asserts CRAP calculation matches hand-computed expected value for 3 functions (low / medium / red band).

That's it. If the tooling grows, so do its tests.

## 12. Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Stryker is slow on 16 files | Scope to `src/lib/**` only; mutation runs in CI-optional `quality:mutation` script, not `quality:all` default. (Re-evaluated if baseline run >5min.) |
| ESLint complexity reporting is function-level but c8 is statement-level | `crap.mjs` maps statement coverage into function ranges using c8's `fnMap` (per-function coverage is in the JSON already). |
| `.mjs` + `.ts` mix confuses tooling | c8 handles both. Stryker needs explicit `mutate: ["src/lib/**/*.{mjs,ts}"]`. Dependency-cruiser needs `--ts-config`. All three verified to work during implementation. |
| Existing `test/*.test.js` files mostly cover `functions/api/`, not `src/lib/` | Expected. The baseline will show near-0% coverage on `src/lib/`. That's the point — it's the starting line. |
| Running Stryker pollutes `src/lib/` with temp files | Stryker's sandbox mode confines writes to `.stryker-tmp/`. Added to `.gitignore`. |

- **Spec C:** Same pipeline applied to `functions/api/` (107 files) — this is where Gherkin/Cucumber is worth adding, because endpoints have user-observable behavior.

## 14. Acceptance Criteria

The pass is done when:

1. ✅ `npm run quality:coverage` produces an HTML report at `reports/quality/coverage/index.html`
2. ✅ `npm run quality:mutation` produces a Stryker HTML report
3. ✅ `npm run quality:crap` produces `reports/quality/crap.md` with every `src/lib/` function listed and banded
4. ✅ `npm run quality:deps` produces `reports/quality/deps.json` and exits cleanly (even if violations found)
5. ✅ `npm run quality:all` regenerates `docs/quality/BASELINE.md` end-to-end
6. ✅ `BASELINE.md` is committed and readable
7. ✅ `reports/quality/` is gitignored
8. ✅ `test/quality-crap.test.js` passes
9. ✅ Running `npm run quality:all` twice in a row produces identical `BASELINE.md` (deterministic)
